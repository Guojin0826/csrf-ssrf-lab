攻击目标：修改用户昵称为"被攻击的用户"

防护机制：CSRF Token验证

💻 攻击代码与防护对比

// 攻击代码（失败）
<!-- 攻击表单缺少有效Token -->
<form id="attackForm" action="csrf_post.php" method="POST">
  <input type="hidden" name="action" value="update_safe">
  <input type="hidden" name="nickname" value="被攻击的用户">
  <input type="hidden" name="phone" value="13800138000">
  <!-- ❌ 缺少有效的csrf_token -->
  <input type="hidden" name="csrf_token" value="fake_token_12345">
</form>

// 防护代码（成功）
// 服务器端验证
if ($_POST['csrf_token'] !== $_SESSION['csrf_token']) {
  die('CSRF Token验证失败，请求被拦截！');
}

// 为什么攻击失败？
1. 攻击者无法获取用户的Session Token
2. 攻击者只能使用伪造的Token（fake_token_12345）
3. 服务器验证Token不匹配，拒绝请求
4. 用户信息未被修改，防护成功

💡 防护原理

CSRF Token是一种服务器生成的随机值，存储在用户Session中。每次表单提交时， 必须携带这个Token。攻击者无法获取用户的Session Token，因此无法构造有效的攻击请求。 即使攻击者伪造Token，服务器验证也会失败，拒绝执行操作。
